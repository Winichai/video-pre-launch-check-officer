# 渲染前预览审批

[使用时机]
    - 0-1 模式：`video-spec.md` 和可选 `design.md` 准备好之后，启动 HyperFrames 之前。
    - 迭代模式：视觉规范、分镜、组件、画面描述、动效、转场或素材依赖发生变化之后。

[目标]
    生成 `preview-board.md`，让用户在渲染前看到每个分镜的关键帧画面、静态画面计划和动态效果计划。
    用户明确批准前，不引导 `/hyperframes`，不启动生成视频。

[输入]
    - 项目根目录的 `video-spec.md`
    - 项目根目录的 `design.md`（如果 §4 视觉规范使用自定义主题）
    - `templates/preview-board-template.md`

[关键帧要求]
    - 如果当前环境能生成或截取图片，必须为每个 Scene 生成至少 1 张关键帧预览图。
    - 关键帧图应保存到项目根目录的 `keyframes/`、`preview-frames/` 或同等清晰命名的文件夹。
    - `preview-board.md` 的每个 Scene 行必须引用对应图片，例如 `![Scene 01](keyframes/scene-01.png)`。
    - 关键帧图用于审批视觉方向，不等于最终视频帧；但构图、字体、颜色、主体元素、重点词和画面密度必须尽量接近最终生成。
    - 如果当前环境不能生成图片，必须在 `preview-board.md` 顶部写明原因，并仍然输出逐镜的文字版关键帧计划。
    - 用户提供了参考图 / 设计范式 / 阶段关键帧时，预览图必须优先贴近这些参考，不要另起视觉系统。

[生成规则]
    - 文件名固定为 `preview-board.md`，放在 `video-spec.md` 同级目录。
    - 每个 Scene 必须有一行，不允许合并多个 Scene。
    - 用人能看懂的话写，不写技术参数，不写毫秒和 easing。
    - 「简单介绍」说明这一镜在讲什么。
    - 「关键帧预览」放图片链接；没有图片时写"未生成，原因：..."。
    - 「预览图计划」说明如果先出一张静帧预览图，它应该长什么样：主体、构图、文字、背景、关键素材。
    - 「动态效果计划」说明画面从开始到结束怎么动：出现、推进、强调、转场。
    - 「风险 / 待确认」只写会导致返工的点：素材缺失、文字太长、画面拥挤、品牌不确定、组件占位、主题冲突。
    - 不要把 `video-spec.md` 原字段机械复制一遍，要翻译成用户能快速判断的视觉预案。
    - 如果某一镜用 `broll-abstract.placeholder`，该行风险必须写明"组件占位，渲染前需确认具体画面"。

[审批规则]
    - 生成 `preview-board.md` 和关键帧图后，明确告诉用户："我先不渲染视频。你先审这个关键帧预览表。"
    - 只有用户明确说"通过 / 批准 / 可以生成 / 开始渲染 / OK 按这个做"时，才算批准。
    - 用户说"看着不错 / 应该可以 / 还行"不算批准，要追问："这是批准生成，还是还要改？"
    - 用户提出修改 → 回到迭代模式，修改 `video-spec.md` 和受影响的 `preview-board.md` 行，再重新等待批准。
    - 用户批准后，不再继续修改 spec；告诉用户可以输入 `/hyperframes` 开始生成视频。

[完成后给用户的话]
    `video-spec.md` 已准备好，`preview-board.md` 也生成好了。
    我先不启动 HyperFrames。你先看关键帧预览表：每一行都是一个分镜，包含关键帧画面、准备做成什么图、会怎么动、可能返工的点。
    你批准后，我再引导启动 `/hyperframes`。
